# iDesign Command Center — Slice 28: Invoice PDF + Email Send (W4) — Design

**Date:** 2026-07-20
**Status:** Approved (design + the pdf-lib dependency); implementation plan pending
**Builds on:** slice 27 (invoices — the sendable artifact), slice 25 (`sendEmail` Resend seam; feature `"invoice"` pre-whitelisted in `EMAIL_FEATURES` since 25-1), slice 24 (audit).
**Unlocks:** slice 29 (payments — the emailed invoice is what gets paid), 30 (history import).

---

## 1. Overview & Goals

Issued invoices become sendable: a server-generated PDF (downloadable at any status, attachable by email for issued only) and a send action stamping `sent_at`/`sent_to`. **Approved dependency: `pdf-lib`** (pure JS/TS, zero native bindings — identical behavior in Netlify functions, vitest, and the D-2 desktop standalone).

**Goals:**
- Model/painter split: `buildInvoicePdfModel` (pure, exhaustively unit-tested) + `renderInvoicePdf` (thin pdf-lib painter, integration-tested by loading bytes back).
- `sendEmail` seam gains optional attachments (additive; simulated mode ignores them).
- Migration `0021`: `invoices.sent_at timestamptz NULL` + `invoices.sent_to text NULL`. NOT a status change — an emailed invoice stays `issued`.
- `sendInvoice({ id, toEmail? })` — issued-only; recipient defaults to frozen `bill_to.email`, per-send override; real (non-simulated) sends stamp sent_at/sent_to; audit verb `"sent"` with the recipient email NEVER in audit/Sentry.
- `GET /invoices/[id]/pdf` — session-guarded route handler, org-scoped, `Content-Disposition: attachment`, DRAFT/VOID banners for non-issued statuses.
- Edit-page UI: Download PDF link (all statuses) + Send panel (issued: prefilled email, send, sent-state line).

## 2. Non-goals (named homes)

Payment links/QR in the PDF → 29. Branded templates/logo upload → module slice. HTML email bodies → polish (plain text is professional + spam-safe). Bulk/scheduled sending, delivery webhooks/read receipts → later. Storing generated PDFs → never v1 (regenerate on demand — the invoice row IS the source of truth; bytes are derived).

## 3. PDF — model/painter split

### 3.1 `src/lib/invoices/pdfModel.ts` (pure)

```ts
export type InvoicePdfModel = {
  banner: "DRAFT" | "VOID" | null;        // issued → null
  header: { orgName: string; title: string; number: string };
  meta: Array<[label: string, value: string]>;   // Issue date / Due date / Status
  billTo: string[];                              // name, business?, email?, address lines (slice-22 shape flattened)
  itemRows: Array<{ description: string[]; qty: string; unit: string; lineTotal: string }>; // description pre-wrapped to maxCharsPerLine
  totals: Array<[label: string, value: string, emphasize: boolean]>;  // Subtotal / Tax (x.xx%) — omitted when 0 bps / Total
  notes: string[] | null;                        // wrapped
  footer: string;                                // "Generated by iDesign Command Center — <UTC date>"
};

export function buildInvoicePdfModel(invoice: InvoiceDetail, orgName: string, now: Date): InvoicePdfModel;
```

All formatting decisions live here: `formatCentsExact` for money, tax label shows the bps as a percent, description wrapping at a fixed char width (pure, deterministic — no font metrics), empty-notes → null. `now` injected.

### 3.2 `src/lib/invoices/pdfRender.ts` (painter)

`renderInvoicePdf(model): Promise<Uint8Array>` — pdf-lib, Letter portrait, Helvetica/Helvetica-Bold standard fonts (no embedding), margins 50pt, y-cursor flow, new page when the cursor crosses the bottom margin (50-item cap keeps this rare), banner drawn large + light-gray behind the header block when present. No logic beyond painting the model.

## 4. Email seam extension — `src/lib/email/` (additive)

```ts
export type EmailAttachment = { filename: string; content: string /* base64 */; contentType: string };
// SendEmailInput gains: attachments?: EmailAttachment[]   (Zod: max 3, filename 1..100, content base64 non-empty ≤ ~7MB base64)
```

Resend body gains `attachments: [{ filename, content }]` when present (Resend infers type; keep contentType in OUR type for future providers). Simulated path ignores attachments entirely. PII rule unchanged: attachment names/content never reach Sentry.

## 5. Send mechanics

### 5.1 Migration `0021` — `invoices` columns

`sent_at timestamptz NULL` (house mode:"date"), `sent_to text NULL`. Additive ALTERs; no index (lookup is always by id).

### 5.2 `sendInvoice` — `src/lib/invoices/actions.ts` (same file, same run() scaffold)

Zod `{ id: int+, toEmail: z.email().max(200).optional() }`. Load org-scoped (missing → Forbidden). **Status must be `issued`** (draft/void → Forbidden — friendly distinct message "Only issued invoices can be sent"). Recipient = `toEmail ?? billTo.email`; neither → `{ ok: false, error: "No email on file for this customer — enter one to send" }`. Fetch orgName (orgs row). Build model (banner null) → render → base64. `sendEmail({ to, subject: \`Invoice ${number} from ${orgName}\`, text: <plain summary: number, total via formatCentsExact, due date if set, item count, "The invoice PDF is attached.">, attachments: [{ filename: \`${number}.pdf\`, content, contentType: "application/pdf" }], feature: "invoice" })`.
- `res.ok && !res.simulated` → stamp `sent_at = now`, `sent_to = recipient` (org-scoped UPDATE). Simulated sends do NOT stamp (the slice-25 precedent — keyless environments must not fake a send record).
- `res.ok && res.simulated` → return `{ ok: true, simulated: true }` so the UI can say "simulated (no email key configured)".
- `!res.ok` → map the seam's error code to a friendly message.
- Audit verb `"sent"` (whitelist append): summary `` `Sent invoice ${number} to ${billTo.name}` ``, payload `{ simulated }` only. Recipient email lives ONLY in the org-scoped `sent_to` column.

### 5.3 `GET /invoices/[id]/pdf` — `src/app/(admin)/invoices/[id]/pdf/route.ts`

Route handler (house API pattern — own session check): `requireSession()` → 401-equivalent redirect/401; demo mode → `getCurrentOrgId()` demo path works (download is pure computation — demo ALLOWED). `getInvoiceById(db, orgId, id)` → null → 404. Any status renders (DRAFT/VOID banner per model). Response: `application/pdf`, `Content-Disposition: attachment; filename="<number>.pdf"` (sanitize the number for the header — quote + strip CR/LF/quotes), `Cache-Control: no-store`.

## 8. Decisions

- PDFs regenerated on demand, never stored — the row is the record, bytes are derived.
- Download allowed at any status (proofreading drafts is legitimate); *sending* is the issued-only act.
- Simulated sends return `simulated: true` and don't stamp — no fake send records, mirrors slice 25.
- Recipient email: `sent_to` column only; never audit/Sentry; shown in-app to the owning org.
- Plain-text email body — professional, spam-safe, zero template debt.
